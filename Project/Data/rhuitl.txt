comment
While the code on this ticket is good, the committed code (in both files) is different:  ``` python     convert_bool = {         'true': True, 'false': False,         'True': True, 'False': False,         '0': False, 0: False,         '1': False, 1: True     } ```  Note the '1': False. 
comment
According to http://docs.oracle.com/javase/6/docs/api/java/lang/Boolean.html#parseBoolean%28java.lang.String%29 Java would parse the string ignoring case (e.g. also "tRuE" is True). The reference to Java is from the non-normative primer though. The normative specification (I could only find XSD files) does not specify the data format for the types. 
comment
I opened a ticket at Boost (https://svn.boost.org/trac/boost/ticket/11092), it really should be fixed there. Meanwhile, I think NetworkX as a library "to get work done" should accept these slightly malformed GraphML documents. It should write perfect ones, of course, but after all it's not a GraphML validator so the benefits of recognizing 0 and 1 will probably outweigh the cons.
